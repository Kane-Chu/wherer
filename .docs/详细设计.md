# Wherer 详细设计文档

## 1. 系统架构概述

### 1.1 设计目标

Wherer 是一款面向个人用户的物品管理助手，核心诉求是"快速记录、随时查找"。用户在日常生活中经常遇到"东西放哪了"的困扰，Wherer 通过"空间-物品"两级结构，将物品与存放位置绑定，实现直观、高效的物品管理。

### 1.2 架构模式：MVVM

本 App 采用 **MVVM（Model-View-ViewModel）** 架构模式，具体实现基于 SwiftUI 的声明式数据流：

- **Model**：由 Core Data 托管对象（Space、Item、ItemPhoto）承担，负责数据持久化与业务规则。
- **View**：纯 SwiftUI 视图，仅负责界面渲染与用户交互响应，不直接操作数据。
- **ViewModel**：由 `SpaceStore`、`ItemStore` 等 ObservableObject 承担，通过 `@StateObject` / `@EnvironmentObject` 注入视图层级，负责业务逻辑编排、数据转换与状态管理。

**选型原因**：SwiftUI 原生拥抱 MVVM，`@Published` 属性与视图自动绑定，可大幅简化 UI 更新逻辑；同时 MVVM 将视图与数据操作解耦，便于后续单元测试与逻辑复用。

## 2. 数据模型设计

### 2.1 实体关系图（ERD）

```

│    Space    │───────────────────────────────│    Item     │

│ id          │                              │ id          │
│ name        │                              │ name        │
│ icon        │                              │ location    │
│ colorHex    │                              │ category    │
│ createdAt   │                              │ tags        │
│ items       │                              │ photoFilename│
└─────────────┘                              │ createdAt   │
                                             │ updatedAt   │
                                             │ space       │
                                             │ photos      │
                                             └──────┬──────┘
                                                    │ 一对多（级联删除）
                                                    │
                                             ┌──────┴──────┐
                                             │  ItemPhoto  │
                                             ├─────────────┤

                                             │ item        │
```

### 2.2 实体说明

#### Space（空间）

空间是物品的一级分类维度，对应现实生活中的物理区域，如"客厅"、"卧室"、"办公室"。

| 属性 | 类型 | 说明 |
|------|------|------|
| id | UUID | 唯一标识符，主键 |
| name | String | 空间名称，用户自定义 |
| icon | String | 图标标识符，用于界面展示 |
| colorHex | String | 主题色十六进制值，用于卡片视觉区分 |
| createdAt | Date | 创建时间 |
| items | NSSet | 关联的物品集合（一对多） |

**设计考量**：
- 采用 `colorHex` 而非直接存储 `Color` 对象，是因为 Core Data 不支持 SwiftUI `Color` 类型，字符串具有更好的跨平台兼容性与序列化能力。
- `icon` 与 `colorHex` 共同构成空间的视觉识别系统，用户可通过图标+颜色快速定位目标空间。

#### Item（物品）

物品是核心管理对象，记录用户需要追踪的具体物品信息。

| 属性 | 类型 | 说明 |
|------|------|------|
| id | UUID | 唯一标识符，主键 |
| name | String | 物品名称 |
| location | String | 具体存放位置描述（如"衣柜第二层"） |
| category | String | 分类标签（如"衣物"、"电子产品"） |
| tags | String | 逗号分隔的标签字符串，用于灵活检索 |
| photoFilename | String | 封面照片文件名（冗余字段，加速列表展示） |
| createdAt | Date | 创建时间 |
| updatedAt | Date | 最后更新时间 |
| space | Space | 所属空间（多对一） |
| photos | NSSet | 关联的照片集合（一对多） |

**设计考量**：
- `photoFilename` 作为冗余字段存储在 Item 中，避免每次加载列表时都要遍历 `photos` 关系查找封面图，提升列表滚动性能。
- `tags` 采用逗号分隔的字符串而非独立 Tag 实体，是因为标签使用场景简单（仅检索），独立实体会增加管理复杂度与同步开销。
- `updatedAt` 用于支持"最近物品"功能，按时间倒序展示用户最近添加或修改的物品。

#### ItemPhoto（照片）

物品的照片附件，支持一张物品关联多张照片，其中一张设为封面。

| 属性 | 类型 | 说明 |
|------|------|------|
| id | UUID | 唯一标识符，主键 |
| filename | String | 照片文件在沙盒中的文件名 |
| isCover | Bool | 是否为封面照片 |
| createdAt | Date | 拍摄/添加时间 |
| item | Item | 所属物品（多对一） |

**设计考量**：
- 照片实体仅存储元数据（文件名），实际图片文件存储在 App 沙盒的文件系统中。Core Data 不适合直接存储大体积二进制数据（BLOB），会导致数据库膨胀、性能下降。
- `isCover` 标记用于快速识别展示用图片，避免在 UI 层遍历全部照片。

### 2.3 级联删除策略

- Space 删除时，级联删除其下所有 Item。
- Item 删除时，级联删除其下所有 ItemPhoto。
- 该策略确保数据一致性，避免产生孤儿记录；同时 PhotoService 需在 Item 删除时同步清理沙盒中的实际图片文件。

## 3. 模块划分

### 3.1 Store 模块（ViewModel 层）

#### SpaceStore

职责：管理空间的生命周期与状态同步。

| 功能 | 说明 |
|------|------|
| 增删改查 | 创建、更新、删除空间；查询全部空间列表 |
| 自动种子数据 | App 首次启动时，若数据库为空，自动创建默认空间（如"客厅"、"卧室"），降低新用户上手门槛 |
| 上下文监听 | 监听 Core Data 上下文保存通知，自动刷新空间列表，确保多线程/多场景下的数据一致性 |

**设计考量**：自动种子数据消除了用户首次打开 App 时面对空白页面的困惑，是提升首次体验的关键设计。

#### ItemStore

职责：管理物品的生命周期、检索与筛选。

| 功能 | 说明 |
|------|------|
| 增删改查 | 创建、更新、删除物品 |
| 搜索过滤 | 接收搜索关键词，返回匹配物品列表 |
| 最近物品 | 按 `updatedAt` 倒序返回最近修改的 N 个物品，用于首页展示 |
| 按空间筛选 | 返回指定空间下的全部物品 |
| 按分类筛选 | 返回指定分类下的全部物品 |

**设计考量**：ItemStore 与 SpaceStore 分离，是因为两者业务逻辑独立、生命周期不同；分离后各自职责单一，便于维护与测试。

### 3.2 Service 模块

#### PhotoService

职责：照片文件的持久化读写。

| 功能 | 说明 |
|------|------|
| 保存照片 | 将 UIImage 写入 App 沙盒的 Photos 子目录，JPEG 格式，质量系数 0.85 |
| 读取照片 | 根据文件名从沙盒加载图片 |
| 删除照片 | 删除沙盒中的图片文件（在物品/照片删除时调用） |

**设计考量**：
- 存储位置选择 App 沙盒（而非 Core Data BLOB），是因为图片文件体积大，文件系统在处理大文件 I/O 时性能远优于数据库存储。
- JPEG 质量 0.85 是在"视觉质量"与"存储空间"之间的平衡：0.85 已接近人眼无损感知阈值，同时文件体积约为原图的 1/5 ~ 1/10。
- 子目录隔离（Photos 目录）便于后续批量清理、备份或迁移。

#### SearchService

职责：提供跨字段的物品检索能力。

| 功能 | 说明 |
|------|------|
| 联合搜索 | 在名称、位置、标签、空间四个字段中同时匹配关键词 |
| 不区分大小写 | 搜索过程忽略大小写，提升用户体验 |

**设计考量**：四字段联合搜索覆盖了用户最可能的检索意图——"我记得名字"、"我记得放哪了"、"我记得标签"、"我记得在哪个空间"；不区分大小写则消除了用户输入时的大小写焦虑。

### 3.3 View 模块（界面层）

视图层级采用 TabView 作为根容器，分为两大主 Tab：

```
ContentView (TabView)
├── SpaceListView (首页 - 空间)
│   ├── SearchBar              # 全局搜索入口
│   ├── SpaceCardView (网格)    # 空间卡片网格展示
│   └── RecentItemsSection     # 最近物品横向滚动区
├── ItemListView (物品页)
│   ├── SearchBar              # 物品搜索
│   └── CategorySection        # 按分类分组的物品列表
│       └── ItemMiniCardView   # 物品缩略卡片
├── ItemDetailView (全屏详情)
│   ├── ImagePreviewView       # 全屏图片预览（点击放大）
│   └── InfoRow                # 信息行组件（位置、分类、标签等）
├── ItemFormView (添加/编辑物品)
│   ├── PhotoPickerPresenter   # 照片选择器封装
│   ├── spacePickerSheet       # 空间选择底部弹窗
│   └── categoryPickerSheet    # 分类选择底部弹窗
└── SpaceFormView (添加/编辑空间)
    └── ColorPresetPicker      # 预设颜色选择器
```

**设计考量**：
- 首页以"空间"为核心组织方式，符合用户"先想地方，再找东西"的心智模型。
- 最近物品区域放在首页顶部，满足高频场景——用户经常找刚放好的东西。
- 物品页以分类为维度，提供另一种浏览路径，适应"不知道放哪了，但知道是什么"的场景。
- 全屏详情页采用沉浸式封面设计，突出物品照片，信息行采用结构化布局，清晰易读。

## 4. 接口设计（组件间交互）

### 4.1 Store 与 View 的交互

Store 通过 `@Published` 属性向 View 发布状态变更，View 通过调用 Store 的公开方法触发业务操作。

**典型交互流程（添加物品）**：

```
1. User taps "+" in ItemListView
2. ItemListView presents ItemFormView
3. User fills form and taps "Save"
4. ItemFormView calls ItemStore.createItem(...)
5. ItemStore creates Core Data object + saves context
6. Core Data triggers context save notification
7. SpaceStore receives notification, refreshes spaces
8. @Published update triggers SwiftUI view re-render
9. ItemListView / SpaceListView automatically reflect new item
```

### 4.2 Store 与 Service 的交互

Store 不直接操作文件系统，而是通过 Service 完成跨层职责。

**典型交互流程（添加物品照片）**：

```
1. User selects photo in ItemFormView
2. ItemFormView passes UIImage to ItemStore
3. ItemStore calls PhotoService.save(image) -> filename
4. PhotoService writes JPEG to sandbox, returns filename
5. ItemStore creates ItemPhoto entity with filename
6. ItemStore saves Core Data context
```

### 4.3 视图间数据传递

- **父子视图**：通过 `@Binding` 或回调闭包传递局部状态。
- **跨层级视图**：通过 `@EnvironmentObject` 注入 Store，避免层层传递。
- **页面跳转**：通过 NavigationLink 或 Sheet 传递实体 ID 或实体对象，目标视图自行从 Store 获取详情。

## 5. 技术选型说明

### 5.1 SwiftUI

- **原因**：Apple 官方推荐的现代 UI 框架，声明式语法与 MVVM 天然契合，开发效率高，且 iOS 17.0+ 已覆盖绝大多数活跃设备。
- **限制说明**：SwiftUI 在部分复杂交互场景（如自定义转场动画）灵活性略低于 UIKit，但本 App 交互复杂度适中，SwiftUI 完全胜任。

### 5.2 Core Data + NSPersistentCloudKitContainer

- **Core Data**：Apple 官方对象图与持久化框架，与 SwiftUI 集成良好（`@FetchRequest`），支持对象关系映射、变更追踪与内存优化。
- **NSPersistentCloudKitContainer**：在 Core Data 基础上自动集成 CloudKit，实现跨设备数据同步。
- **失败回退本地**：若用户未开启 iCloud 或网络异常，自动回退到本地 SQLite 存储，确保 App 始终可用。

**选型原因**：该组合是 Apple 生态内"本地持久化 + 云端同步"的标准方案，无需自建后端，数据完全由用户掌控，隐私合规成本低。

### 5.3 沙盒文件系统存储照片

- **原因**：Core Data 存储大体积图片会导致数据库性能劣化；沙盒文件系统专为文件存储优化，支持流式读写。
- **路径选择**：存储在 App 沙盒的 Documents/Photos 子目录下，该目录会被 iCloud 备份（若用户开启），同时受 iOS 沙盒机制保护，其他 App 无法访问。

## 6. 安全与隐私设计

### 6.1 数据隔离

- 所有用户数据（Core Data 数据库、照片文件）均存储在 App 沙盒内，iOS 沙盒机制确保其他 App 无法访问。
- 不收集任何用户行为数据或崩溃日志到第三方服务器。

### 6.2 云端同步安全

- 通过 CloudKit 同步时，数据经 Apple 加密通道传输，存储在用户的私有数据库中。
- CloudKit 的私有数据库仅用户本人可访问，App 开发者、Apple 均无法读取。

### 6.3 照片隐私

- 照片访问通过系统 `PHPickerViewController`（或 SwiftUI `PhotosPicker`），用户自主选择照片，App 不请求完整的照片库权限。
- 用户选择的照片被复制到 App 沙盒后，原照片库中的图片不受影响。

### 6.4 数据备份

- 开启 iCloud 备份的用户，App 沙盒内容会自动纳入 iCloud 设备备份。
- Core Data + CloudKit 的组合本身也提供了跨设备实时同步能力，作为备份的补充。

## 7. 性能设计要点

| 场景 | 策略 | 原因 |
|------|------|------|
| 空间/物品列表加载 | 使用 `@FetchRequest` 或 Store 预加载，配合 SwiftUI 懒加载 | 避免一次性加载大量数据 |
| 照片列表展示 | 封面图使用 `photoFilename` 冗余字段，避免遍历 photos 关系 | 减少关系查询开销 |
| 图片文件读写 | 异步执行，JPEG 质量 0.85 | 避免阻塞主线程，平衡质量与体积 |
| 搜索响应 | SearchService 在内存中过滤，关键词变化防抖处理 | 减少频繁查询对数据库的压力 |
